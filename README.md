# 手握罗盘的舵手

> 基于数据思维和系统思维，让非技术背景的人也能把控AI编程

---

## 你是否经历过这些困境？

你是一个产品经理、设计师、创业者，或者只是有一个想法想要实现。你听说AI可以帮忙写代码，于是兴冲冲地开始"Vibe Coding"——凭感觉和AI对话，希望它能神奇地把你的想法变成现实。

但很快，你遇到了这些痛点：

| 痛点 | 你遇到的情况 | 为什么会这样 |
|------|-------------|-------------|
| **黑盒焦虑** | AI说"我已经完成了"，但你完全不知道它做了什么，改了哪些文件，会不会把其他东西弄坏？ | AI的思考过程对你不可见，你只能被动接受结果 |
| **零碎协作** | 来一个问题改一个问题，AI不知道全局，经常改了A坏了B，陷入无休止的"修复-再修复"循环 | 缺乏系统化视角，每次修改都是孤立的操作 |
| **进度迷雾** | 问AI"项目进展如何"，它只能给你一个模糊的回答，你不知道真正的完成度，也无法识别风险 | 没有数据支撑，协作状态完全依赖AI的主观判断 |
| **重复返工** | 你明确说过"这个功能我很满意"，但AI重构时又把它改了，你的反馈没有被记住 | 人类的反馈没有被系统化记录，AI无法识别"已满意"的边界 |
| **能力天花板** | 每次只能让AI做一件事，遇到复杂问题需要多步协作时，AI就"失忆"了 | AI的能力是原子化的，无法组合成更强大的系统级能力 |
| **知识流失** | 项目做完了，但在这个过程中积累的经验、规则、最佳实践都散落在对话历史中，无法复用 | 协作过程中的隐性知识没有被沉淀和结构化 |

---

## 我们的答案：确定性AI协作模式

**核心主张**：AI协作不应该是"感觉对了就行"的黑盒游戏，而应该是**透明可见、数据驱动、系统化运作**的确定性过程。

### 三个核心原则

#### 1. 透明化协作：打破黑盒

```
传统Vibe Coding          确定性协作
┌─────────────┐          ┌─────────────────────┐
│  你: 做这个  │          │  AI: 我观察到项目状态  │
│      ↓       │          │      数据呈现给你      │
│  AI: 好的    │    vs    │  你: 确认/调整方向    │
│      ↓       │          │      ↓               │
│  ???黑盒???  │          │  AI: 执行并实时观测   │
│      ↓       │          │      ↓               │
│  AI: 完成了  │          │  AI: 效果验证+改进建议 │
│      ↓       │          │      ↓               │
│  你: 哎坏了   │          │  你: 确认/继续优化    │
└─────────────┘          └─────────────────────┘
```

**怎么做？**
- AI帮你打开终端，请求你测试，AI同时观测运行过程
- 实时向你展示测试结果和改进建议
- 所有决策基于实际运行数据，而非猜测

#### 2. 数据驱动规则：只有基于真实数据的规则才有效

很多人在开始AI协作前就制定了一大堆规则：
- "代码风格要统一"
- "函数不超过50行"
- "必须有完整的注释"

**但这些规则往往无效**——因为它们不是基于你的项目实际运行数据分析得出的。

**有效规则的特征**：
```
空洞规则（无效）              数据驱动规则（有效）
─────────────────          ─────────────────
"代码要快"         →        "登录接口响应时间>800ms，
                           需要优化数据库查询"

"不要出bug"       →        "测试数据显示用户注册流程
                           有15%失败率，需要添加
                           支付渠道超时处理"

"代码要清晰"       →        "性能追踪显示checkout()
                           函数耗时占比60%，需要拆分"
```

**怎么做？**
- 帮你自动采集数据
- 基于真实的性能指标、错误率、用户行为数据制定规则
- 针对你的项目运行数据进行的优化和重构才真正有效

#### 3. 系统化人机共识：可视化看板替代零碎操作

```
传统零碎协作                系统化共识协作
─────────────────          ─────────────────
来一个问题改一个问题   →   AI主动请求你对功能
                           完成度的反馈
                           ↓
不知道改了什么影响了什么   →   批量制定改进计划
                           可视化看板展示
                           ↓
修复时引入新bug         →   重构时自动绕开你已
                           满意的功能
                           ↓
无法形成全局认知         →   人机达成真正的共识
```

**怎么做？**
- AI主动请求你对每个功能的完成度反馈（满意/需改进）
- 批量生成改进计划，而不是零散地一个问题接一个问题
- 可视化看板展示项目全局状态
- 重构时自动识别并保护你已经满意的功能

## 技能体系：自我组合与自我进化

这套技能不只是四个独立工具，它们可以**自动组合**成更强大的系统级能力，并能**自我进化**。

### 技能自我组合：1+1>2

```
┌─────────────────────────────────────────────────────────┐
│                    技能组合能力示例                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   code-observer + dev-observability                     │
│   = 代码性能瓶颈自动识别 + 项目进度关联分析               │
│                                                         │
│   dev-observability + skill-evolution-driver            │
│   = 项目风险自动预警 + 技能自动优化触发                  │
│                                                         │
│   code-observer + skill-evolution-driver                │
│   = 代码问题自动发现 + 技能自我修复建议                  │
│                                                         │
│   四个技能协同 = 确定性协作闭环                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 技能自我进化：随着项目成长

技能不是静态的工具，它们会随着项目进展**自动分析需要完善的地方**：

```
项目启动阶段                  项目迭代阶段                  项目成熟阶段
─────────────                ─────────────                ─────────────
发现缺少进度跟踪功能    →     自动触发添加测试埋点能力   →   积累项目专属的
自动建议启用dev-observability  自动优化代码分析规则        知识库和规则集
```

**私域知识沉淀**：
- 随着技能的自我成长，你的项目专属知识（架构决策、性能基线、错误模式）也在沉淀
- 这些知识会成为技能未来的决策依据，让AI越来越懂你的项目

## 四大技能一览

| 技能 | 角色 | 核心价值 |
|------|------|---------|
| **skill-manager** | 统一存储服务 | 为所有技能提供数据存储和共享能力，是技能协作的基础设施 |
| **dev-observability** | 开发可观测底座 | 项目/应用/测试三维度透明化，让协作状态实时可见 |
| **code-observer** | 代码调试智能助手 | 追踪代码执行路径，发现性能瓶颈，定位错误根因 |
| **skill-evolution-driver** | 技能进化驱动师 | 自动发现技能优化机会，执行安全更新，确保技能持续改进 |

### 协作关系图

```
                    ┌─────────────────────┐
                    │ skill-evolution-    │
                    │     driver          │
                    │   (自我进化驱动)     │
                    └──────────┬──────────┘
                               │ 监控 & 优化
                               ▼
┌──────────────────┐     ┌─────────────────────┐
│  code-observer   │────▶│  skill-manager      │
│  (代码透明化)     │     │  (统一数据存储)      │
└──────────────────┘     └──────────┬──────────┘
          读取                    ▲

```

## 跨平台兼容

本技能体系采用标准化的技能格式（YAML前置元数据 + Markdown文档），已测试兼容主流AI平台的技能系统：

| 平台 | 兼容性 | 说明 |
|------|-------|------|

### 操作系统兼容性

- **Windows**：完全支持（MSYS/Git Bash/WSL）
- **macOS**：完全支持（原生终端）
- **Linux**：完全支持（原生终端）

所有Python脚本均已进行跨平台路径处理兼容。

## 怎么使用

### 前置要求

- 支持技能的AI平台（Claude Code/Coze/Dify等）
- Python 3.7+ (部分技能需要)
- Flask 2.0+ (dev-observability Web 方案)

### 安装步骤

1. **将技能目录复制到平台的技能目录**
   - Claude Code: `~/.claude/skills/` 或 `.claude/skills/`
   - Coze: 通过平台界面导入技能包
   - 其他平台: 参考相应平台的技能安装指南

2. **重新启动AI平台以加载技能**

### 各技能详细使用说明


#### dev-observability - 开发可观测底座

**触发时机**：
- 项目启动时，需要建立项目监控
- 你想了解项目进展、识别风险时
- 出现复杂问题或重复问题，需要系统性跟踪时

**触发方式**：

| 触发词类别 | 具体触发词 |
|-----------|-----------|
| **核心理念触发** | "透明化"、"确定性协作"、"观察项目情况"、"启用观察"、"了解系统健康状态" |
| **项目进展查询** | "想知道项目进展如何"、"有没有延期风险"、"迭代完成度是多少" |
| **问题追踪** | "经常出bug怎么跟踪"、"重复问题怎么处理"、"需要记录待解决的问题" |
| **质量保障** | "需要监控接口性能"、"怎么发现系统异常"、"测试覆盖率怎么样" |
| **团队协作** | "需要共享项目信息"、"让团队成员了解现状"、"需要可视化仪表板" |

**典型对话示例**：

```
你: 启用观察
AI: [检测到dev-observability触发]
    帮你建立项目可观测性。首先确认几个问题：
    1. 你的团队规模？（个人/小团队/大团队）
    2. 需要什么级别的可视化？（简单日志/Web界面/完整监控）
    ...
```

**部署方案选择**：
- **简单方案**：仅生成结构化日志，适合个人开发
- **中等方案**：提供本地Web界面，适合小团队（推荐）
- **全面方案**：Prometheus+Grafana专业监控（待上线）

#### code-observer - 代码调试智能助手

**触发时机**：
- 代码运行出现性能问题（太慢）
- 代码运行出现错误，不知道哪里出问题
- 业务逻辑复杂，理不清楚执行顺序

**触发方式**：

| 触发词类别 | 具体触发词 |
|-----------|-----------|
| **性能问题** | "代码跑得慢"、"这个函数为什么会这么慢"、"性能瓶颈在哪里" |
| **错误排查** | "代码运行到一半就报错了"、"不知道哪里出问题"、"定位bug" |
| **逻辑理解** | "这个业务逻辑太复杂了"、"理不清楚执行顺序"、"想看看代码是怎么执行的" |
| **代码质量** | "观察代码运行情况"、"代码质量"、"优化代码" |

**典型对话示例**：

```
你: 我想看看这个函数为什么会这么慢
AI: [检测到code-observer触发]
    我来帮你追踪代码执行。首先确认调试环境：
    1. 请打开应用终端（如果还没打开）
    2. 确认dev-observability的日志正在记录
    ...

    [读取日志并分析后]
    发现性能瓶颈：checkout()函数耗时占比60%
    原因：数据库查询未使用索引
    建议：[具体优化方案]

    另外，在分析中发现一个技能优化机会：
    dev-observability的日志格式缺少timestamp字段
    是否需要调用skill-evolution-driver处理这个优化？
```

**重要特性**：
- 会自动调用skill-manager查询数据存储位置
- 发现技能优化点时会主动询问是否触发skill-evolution-driver
- 依赖文件缺失时强制记录问题到skill-manager

---

#### skill-evolution-driver - 技能进化驱动师

**触发时机**：
- 你直接要求优化或改进技能
- code-observer发现技能问题并询问你是否优化
- 定期监控发现技能有优化机会（自动触发提醒）

**触发方式**：

| 触发词类别 | 具体触发词 |
|-----------|-----------|
| **用户主动需求** | "优化技能"、"改进技能"、"技能需要更新"、"技能升级"、"修复技能问题" |
| **版本管理** | "技能版本号需要更新"、"需要记录技能变更" |
| **质量检查** | "技能格式不正确"、"需要检查技能质量"、"技能测试不通过" |
| **自动触发** | skill-manager存储的数据中检测到待优化任务、错误日志、缺失字段 |

**典型对话示例**：

```
你: 优化技能
AI: [检测到skill-evolution-driver触发]
    [检查skill-manager存储的数据]

    检测到以下优化机会：

    1. 技能：dev-observability
       - 优化类型：脚本输出优化
       - 问题：observability.log文件缺少timestamp字段
       - 影响：无法按时间顺序追踪代码执行路径

    2. 技能：code-observer
       - 优化类型：格式改进
       - 问题：SKILL.md缺少version字段
       - 影响：版本管理不规范

    是否开始优化？（可选择性处理）
```

**进化流程**：
1. 提醒用户优化机会
2. 维护优化任务清单（存储在skill-manager）
3. 分析可行性
4. 备份技能
5. 执行优化并更新版本号
6. 测试验证
7. 测试通过则完成，不通过则自动还原
8. 通知用户结果

---

#### skill-manager - 技能管理者

**触发时机**：被`dev-observability`、`code-observer`、`skill-evolution-driver`等其他技能自动调用

启用观察          # 启动dev-observability，建立项目可观测性
